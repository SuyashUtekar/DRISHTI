This document details every step, sub-step, decision, and optimization in the end-to-end agentic pipeline — from raw natural language query to synthesized, audit-logged response. Each component includes what it does, how it does it, and the time complexity improvement applied. The system achieves ~71% latency reduction through parallel execution, smart caching, and rule-first fallback patterns.

Foundation Layer
Data Creation
1.a DDL Schema Creation One-time
  • Create PostgreSQL database rbi_supervision_db with UTF-8 encoding
  • Define 7 SQL tables in FK-safe seeding order: bank_entities → rbi_circulars_registry → npa_accounts → bank_inspection_reports → penalty_orders → digital_payment_metrics → fraud_reports
  • Apply CHECK constraints on all enum-type columns (bank_type, npa_category, status)
  • Add GENERATED ALWAYS AS column for reporting_delay_days in fraud_reports — auto-computed from detected_date and reported_date
  • Create 3 materialized views: v_bank_risk_summary, v_fraud_delay_violations, v_uncomplied_penalties
  • Create MongoDB collections: bank_complaints, supervisory_alerts, bank_returns_submissions with JSON schema validators
  • Create compound indexes on all foreign key + filter columns for sub-10ms query execution
1.b.i SQL Metadata Store Cached
  • Store complete schema metadata in a dedicated sql_metadata table: table name, column names, data types, nullable flags, PII flags, FK relationships
  • Pre-compute join map — which tables can be joined on which keys: fraud_reports.account_id → npa_accounts.account_id, *.bank_id → bank_entities.bank_id
  • Store column statistics: min/max values, distinct count, null percentage — used by NL-SQL agent to write precise WHERE clauses
  • Tag each column with department access: which dept can query which column
  • Refresh metadata cache every 6 hours via cron — not per query
  • Load full metadata into Redis at startup for 0ms agent access
Cache SQL metadata at startup in Redis — saves 80-120ms per query vs fresh DB fetch
1.b.ii NoSQL Metadata Store Cached
  • Store collection metadata: collection name, top-level field names, nested field paths, field types, example values
  • Pre-compute field cardinality for filter optimization — e.g. status field has 4 distinct values
  • Store index definitions — which fields are indexed for fast filter queries
  • Map soft FK references: bank_id in all 3 collections references bank_entities.bank_id in Postgres — documented for cross-store join planning
  • Cache in Redis alongside SQL metadata at startup
1.b.iii Document / Vector Metadata Store Cached
  • Register all 43 documents in a metadata index: doc_id, title, circular_ref, bank_id (if bank-specific), chunk count, page count, topic tags, dept_relevance
  • Store pre-generated summaries for each document as a dedicated executive_summary chunk — retrieved first for overview queries
  • Store image alt-text summaries as text chunks alongside parent document — enables visual content to be retrieved via semantic search
  • Store OCR output of all tables as structured JSON/Markdown — permanently cached, never re-computed
  • Map each document to its SQL counterpart: AFI_BNK002_PNB_2023 ↔ INS-PNB-2023 in inspection_reports table
  • Pre-cluster topics using k-means on embeddings — 8 topic clusters for fast coarse filtering before dense search
Pre-cluster topics at index time — reduces dense search space from 1,154 to ~140 chunks per query

  • Pre-execute all 50 sample queries and cache results with tiered TTL strategy

3
Gate Layer · Tool
Intent Classifier Tool — L1 (Valid / Invalid)
3.a Layer 1 — Regex Rule Engine 0–2ms
  • Apply compiled regex patterns for SQL injection: DROP|DELETE|UPDATE|INSERT|TRUNCATE|ALTER|EXEC
  • Block sensitive credential queries: password|secret|api_key|token
  • Block XSS patterns: <script|javascript:|onerror=
  • If any pattern matches → immediately return INVALID with reason — no LLM call needed
3.b Layer 2 — Structural Checks 2–5ms
  • Length check: reject if len(query) < 5 or len(query) > 1000
  • Language detection: reject if query is not English or Hindi (RBI use case)
  • Character set check: reject excessive special characters suggesting injection
  • If structural check fails → return INVALID with reason
3.c Layer 3 — Domain Keyword Check 5ms
  • Check presence of RBI domain keywords: bank, npa, fraud, loan, penalty, circular, complaint, inspection, camels, rbi, dpd, sarfaesi, upi, kyc, fmrs
  • If at least 1 domain keyword found → return VALID immediately — no LLM call
  • If no domain keyword found → escalate to LLM fallback layer
Skip LLM for 80%+ of valid banking queries — saves 150ms on every standard query
3.d Layer 4 — LLM Fallback (only if layers 1-3 inconclusive) 150ms
  • Call lightweight LLM (GPT-3.5-turbo or Claude Haiku) with a strict yes/no validation prompt
  • Prompt: "Is this query related to RBI bank supervision, financial regulation, or banking data? Answer YES or NO only."
  • Return VALID or INVALID based on response
  • Log every L4 call — if many reach L4, expand domain keyword list
4
NLP Layer · Parallel
Query Decomposition and Paraphrasing
Run 4.a and 4.b in parallel using asyncio.gather() — total time = max(decompose, paraphrase) not sum
Parallel Execution Block
4.a — Query Decomposition (NLP + SpaCy)
  • Load SpaCy model once at startup as singleton — never reload per query
  • NER: extract entities (bank names, amounts, dates, DPD values, city names)
  • Dependency parsing: identify subject (what) + predicate (action) + object (filter)
  • Intent extraction: detect if query has multiple sub-intents — split into sub-queries
  • Example: "fraud cases with delay AND unpaid penalties" → Sub-Q1: fraud delay query + Sub-Q2: penalty query
  • Output: structured intent object with entities, filters, sub-queries, join hints
4.b — Query Paraphrasing (SLM)
  • Use quantized flan-t5-small (INT8) — 80ms vs 600ms for large model
  • Generate 2 paraphrases to increase semantic search coverage
  • Paraphrase 1: formal version — for vector DB search
  • Paraphrase 2: keyword-dense version — for BM25 pre-filter
  • Example original: "show me bad loans in PNB" → Para1: "List NPA accounts classified at Punjab National Bank" → Para2: "NPA DPD overdue loan PNB accounts"
  • All 3 versions (original + 2 paraphrases) used in vector retrieval for higher recall

Routing Layer · Tool
Intent Classifier Tool — L2 (Source + Query Type)
5.a Routing Cache Check — Before LLM 0ms
  • Generate query fingerprint: lemmatize all non-stop tokens → sort → hash
  • Look up fingerprint in routing cache stored in Redis
  • Cache hit → return source + query_type + tool_plan immediately — skip LLM entirely
  • Cache miss → proceed to signal-based routing rules
Cache hit rate ~45% for RBI queries — officers ask similar questions repeatedly
5.b Signal-Based Rule Routing 5ms
  • Apply keyword signal rules to decomposition output — no LLM needed
IF signals: count, sum, list, filter, average, DPD, outstanding, penalty_amt → Route: SQL | Type: report/lookup
IF signals: complaint, alert, event, submission, activity, log → Route: NoSQL | Type: lookup/summary
IF signals: circular, says, document, report says, what does, guidelines → Route: Vector | Type: lookup/explanation
IF signals: show chart, visualize, trend, month-wise, compare → Type: visualization (+ SQL/NoSQL for data)
IF multiple entity types from decomposition (3+ distinct source signals) → Route: Multi-hop (all sources) | Type: multi_hop_report
5.c Single LLM Call — Source + Type + Tool Plan 150ms
  • Only called if signal routing is inconclusive (ambiguous query)
  • Single structured output call — returns source + query_type + tools_needed + parallel_executable in one JSON response
  • Inject decomposition output + source registry metadata as context — not raw query alone
  • Output includes: sources[], query_type, tools_needed[], parallel_executable: bool, confidence: float
  • Store result in routing cache with 300s TTL for future similar queries
Combined L2 + tool plan in one LLM call — eliminates a second LLM call that was previously separate
6
Data Tool · SQL
NL-SQL Tool (LLM)
6.a Prompt Construction 5ms
  • System prompt: "You are an RBI database analyst. Generate only valid PostgreSQL SELECT statements. Never generate INSERT, UPDATE, DELETE, DROP, ALTER, TRUNCATE, GRANT. Always include LIMIT clause. Use only column names from the provided schema."
  • Inject relevant table schemas from Redis metadata cache — only tables needed for this query, not all 7
  • Inject join map for tables involved — pre-computed, not described in prose
  • Inject department access rules: which fields are masked for this user's department
  • Inject decomposition output: extracted entities, filters, and sub-queries
  • Include 2-3 few-shot SQL examples similar to the current query type
6.b SQL Generation + Pre-execution Guard Rails 200ms
  • LLM generates parameterized SQL string
  • Parse SQL AST using sqlparse before touching the database — 2ms
  • Guard 1: Block if statement type is not SELECT — reject immediately
  • Guard 2: Scan all tokens for blocked keywords — reject if found
  • Guard 3: Verify all table names exist in schema metadata — reject hallucinated tables
  • Guard 4: Verify all column names exist in respective table schemas — reject hallucinated columns
  • Guard 5: Auto-inject LIMIT 500 if no LIMIT clause present
  • Guard 6: Strip denied PII fields from SELECT list based on dept access rules
  • Only if all guards pass → execute via MCP connector on read-only DB connection pool
Pre-parse AST before execution — blocks bad SQL at 2ms, never hits the database
6.c Execution + Result Packaging 50-100ms
  • Execute via asyncpg connection pool (min 5, max 20 connections) — no connection overhead
  • Return rows as list of dicts with column names preserved
  • Tag result with: source=SQL, table_names, row_count, execution_time_ms, query_hash
  • Write tool call to audit buffer (async — does not block result return)
7
Data Tool · NoSQL
NL-NoSQL Tool (LLM)
7.a Prompt Construction 5ms
  • System prompt: "You are an RBI MongoDB analyst. Generate only valid MongoDB find() or aggregate() queries. Never generate insert, update, delete, drop, or createCollection operations. Always include projection to limit fields returned."
  • Inject collection schema: top-level fields, nested paths, field types, example values from metadata cache
  • Inject index definitions — guide LLM to use indexed fields in filters for performance
  • Inject soft FK map — which bank_id values are valid (from bank_entities SQL table)
  • Include 2 few-shot MongoDB query examples for complaint and alert collections
7.b Query Generation + Guard Rails 200ms
  • LLM generates MongoDB query as JSON object
  • Guard 1: Parse JSON and check operation type — only find and aggregate allowed
  • Guard 2: Scan for blocked operators: $out, $merge, $lookup with write — reject if found
  • Guard 3: Verify all field names in filter exist in collection schema
  • Guard 4: Auto-add projection to exclude PII fields not accessible to current department
  • Guard 5: Auto-inject .limit(200) if no limit present
  • Execute via AsyncIOMotorClient connection pool — non-blocking
7.c Result Packaging 10ms
  • Serialize BSON documents to plain Python dicts — strip ObjectId fields
  • Tag result with: source=NoSQL, collection_name, doc_count, execution_time_ms, query_hash
  • Write audit entry async — non-blocking
8
Data Tool · Vector
Document / Vector Search Tool (VLM + LLM)
8.a Two-Stage Retrieval — Text Chunks with Embeddings 30ms total
  • Stage 1 — BM25 Pre-filter (10ms): Run keyword-based BM25 search on all 1,154 chunks using keyword-dense paraphrase — narrows to top 50 candidate chunks
  • Stage 2 — Dense Vector Search (20ms): Embed original query + formal paraphrase using BAAI/bge-m3 — run dense similarity search on 50 BM25 candidates only, not full index
  • Use embedding cache: if this query text was embedded before → return cached vector, skip embedding model — saves 80ms
  • Apply metadata pre-filter before vector search: dept_relevance must include current user's department
  • Apply circular_ref pre-filter if query mentions a specific circular number
  • Return top-10 chunks ranked by hybrid score: 0.4 × BM25_score + 0.6 × cosine_similarity
Two-stage retrieval: 10x smaller search space for dense retrieval — saves 90ms vs full index scan
8.b Tabular Data — OCR → Markdown / JSON Cached forever
  • Check OCR cache first using doc_id + page_number as key — OCR output never changes for static documents
  • Cache hit → return Markdown table / JSON instantly — 0ms OCR cost
  • Cache miss → run OCR pipeline: pdf2image → pytesseract → table detection → Markdown/JSON serialization — 2000-5000ms
  • Store OCR result permanently in Redis with no expiry
  • For CAMELS tables: serialize as structured JSON with component names and scores — queryable by validator
  • For NPA distribution charts: serialize alt_text_summary as text chunk + structured_data as JSON
Permanent OCR cache — first run is slow (2-5s), every subsequent query is instant (0ms)
8.c Image Summary — VLM (Vision Language Model) Only on cache miss
  • Check image summary cache using image_id as key — static documents never change
  • Cache hit → return pre-generated alt_text_summary as text chunk — 0ms
  • Cache miss → send image to VLM (GPT-4V or Claude 3): "Describe this chart/table from an RBI bank inspection report. Include all numbers, percentages, bank names, and key findings."
  • VLM generates structured summary including: chart type, all data points, key insight, risk signal if present
  • Store VLM-generated summary permanently — never re-run for same image
  • Embed summary as a new chunk and add to vector index — future queries can retrieve it semantically

Output Tool
Visualization Tool
9.a Chart Type Inference 2ms
  • Infer chart type from query_type and data shape using rule map:
IFtime-series data (month column present) + numeric metric→ Line chart
IFcategorical comparison (bank names + single metric)→ Bar chart
IFpart-of-whole (sector distribution, NPA breakdown)→ Pie / Donut chart
IFmulti-metric comparison across banks→ Grouped bar / Radar chart
IFtabular data with no clear visual structure→ Formatted table (no chart)
9.b Client-Side Spec Generation (not server render) 5ms
  • Generate Chart.js or Vega-Lite specification JSON — do not render to PNG/PDF on server
  • Stream chart spec to frontend immediately — frontend renders in browser (client-side)
  • Perceived render time: ~50ms in browser vs 800-2000ms for server-side image generation
  • Include RBI color palette in spec: risk levels (RED=critical, AMBER=watch, GREEN=healthy)
  • Include data annotations: flag values that exceed RBI thresholds with visual markers
Client-side rendering saves 800ms+ — spec is 1KB JSON vs 200KB server-rendered PNG

Quality Gate · Tool
Validator Tool
10.a Rule-Based Validation — All Checks (run first) 10ms
  • Rule 1: Row count non-zero — if 0 rows returned, flag as NO_DATA with suggestion to broaden filter
  • Rule 2: Schema type check — all output field types match expected types from metadata
  • Rule 3: PII field check — scan output for Aadhaar pattern (12 digits), PAN pattern (ABCDE1234F) — block if found
  • Rule 4: Bank ID referential check — all bank_ids in output exist in bank_entities master table
  • Rule 5: Amount sanity check — no amounts below ₹0 or above ₹10,00,000 Cr (likely data error)
  • Rule 6: Date range check — no future dates in historical data fields
  • Rule 7: Cross-source consistency — if same bank_id appears in SQL + NoSQL results, bank_name must match
  • If all 7 rules pass → confidence = 0.95, skip LLM validator entirely
Rule-first validation: 10ms vs 150ms LLM call — use LLM only when rules detect anomaly
10.b Parallel Execution with Synthesizer Parallel
  • Start validation and synthesis simultaneously using asyncio.gather()
  • If validation passes (which it does 90%+ of the time) → synthesis draft is already ready — zero additional wait
  • If validation fails → apply corrections to synthesis draft — adds ~50ms fix time
  • LLM validator called only for anomalous results — hallucination check on borrower IDs, amount plausibility, source traceability
Parallel validation + synthesis — saves 200-300ms for every query

Orchestration · Agent
Planner Agent
11.a Plan Cache Check 0ms
  • Generate plan cache key from: query_type + sorted(sources)
  • Look up in plan cache — pre-built plans for known query patterns
  • Cache hit → return tool execution plan instantly — skip LLM planning entirely
  • Known plan patterns: fraud_delay_report → [nl_sql, validator, viz] | circular_lookup → [vector_search, synthesizer] | full_risk_dashboard → [nl_sql, nl_nosql, vector_search, validator, viz, synthesizer]
11.b LLM Planning (only for novel query patterns) 200ms
  • Prompt includes: L2 classification output, decomposition output, available tools with docstrings, dept access rules, parallel execution capability flag
  • LLM outputs: ordered list of tool calls with parameters, which steps can run in parallel, expected output format for each step, fallback plan if a tool fails
  • Tool docstrings injected into prompt — LLM knows exact input/output contract of each tool
  • Output plan is a DAG (Directed Acyclic Graph) — nodes are tool calls, edges are data dependencies
  • Store new plan in plan cache with 600s TTL
11.c Context Assembly 5ms
  • Assemble full context for executor: query, decomposition, paraphrase, L2 classification, plan, dept rules, session short-term memory
  • Apply context window budgeting: allocate token budget per tool — SQL schema gets 800 tokens, vector context gets 1200 tokens
  • Load relevant short-term memory: last 3 turns of conversation for follow-up awareness
  • Load relevant long-term memory: domain glossary terms found in query, applicable RBI norms
12
Execution · Agent
Executor Agent
12.a DAG Execution — Parallel Where Possible 300ms (parallel)
  • Parse the planner's DAG — identify which tool calls have no dependencies on each other
  • Fire all independent tool calls simultaneously using asyncio.gather()
  • Example for multi-hop query: NL-SQL + NL-NoSQL + Vector Search all fire at same time — total time = slowest single call (~300ms), not sum (~900ms)
  • Dependent tool calls (e.g. join step depends on both SQL and NoSQL results) wait only for their specific dependencies
  • Pass results through connection pool — never open new DB connections mid-execution
Parallel execution of SQL + NoSQL + Vector — single biggest latency win: 600ms saved on every multi-hop query
12.b Error Handling + Fallback Execution On failure only
  • If a tool call returns an error → executor does not fail the whole pipeline
  • Timeout threshold: 800ms per individual tool call — if exceeded, mark as TIMEOUT and continue with available results
  • SQL returns 0 rows → executor automatically escalates to vector search for same entity — checks if answer exists in documents
  • Tool failure logged to audit trail — validator notified to flag confidence as LOW for missing source
  • If 2+ tools fail → executor escalates to planner for re-planning with remaining tools
12.c Result Collection + Multi-Hop Join 20ms
  • Collect results from all completed tool calls into a unified result dict keyed by source type
  • For multi-hop queries: perform in-memory inner join on bank_id or account_id — Python pandas merge or simple dict intersection
  • Attach source provenance to each data point — every row knows which tool/source it came from
  • Store complete result set in short-term session memory — enables follow-up queries without re-execution
  • Write full tool call audit log async: tool_name, input_params, output_row_count, latency_ms, timestamp

Output · Agent
Synthesizer Agent — Final Output
13.a Response Synthesis — Streaming Output 200ms to first token
  • Start streaming synthesis simultaneously with validation — user sees first tokens while validation is still running
  • Prompt: inject validated results, domain glossary context, query intent, output format requirement (summary/report/visualization/explanation)
  • Apply domain language: replace technical DB values with RBI terminology — DPD > 90 → "NPA Sub-Standard per IRACP norms"
  • Structure output by query type: lookup → prose paragraph | report → table + summary | multi-hop → findings + table + risk flags | visualization → chart spec + narrative
  • Cite every data point with source: "According to core_banking_pg — fraud_reports table"
  • Stream tokens to UI using Server-Sent Events — perceived latency = 200ms to first visible character
Streaming output: perceived latency drops from 2000ms (full response wait) to 200ms (first token)
13.b Confidence Score + Source Citations 5ms
  • Append confidence score at end of response: HIGH (0.9+), MEDIUM (0.7-0.9), LOW (<0.7)
  • Confidence derived from: validation result + source count corroborating the answer + row count returned
  • List all sources queried: table names, collection names, document IDs
  • For LOW confidence: append disclaimer "Recommend manual verification before action"
13.c Memory Update + Audit Log Async — non-blocking
  • Store result summary in short-term session memory — enables follow-up: "show only HIGH risk from above", "export this to PDF"
  • Update long-term query pattern library: if this query type is new, add plan as new pattern for future reuse
  • Write immutable audit log entry: session_id, user_dept, query_hash, tools_called, sources_accessed, row_counts, pii_masked, confidence, total_latency, data_lineage_chain, timestamp
  • Audit write is async and append-only — does not block response delivery to user
  • Proactive suggestion: based on result, suggest 1 follow-up query — "Want me to show the trend for these banks over the last 4 quarters?"
