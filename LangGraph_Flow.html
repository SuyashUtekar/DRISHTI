Node Steps
  • Regex Layer (0ms): Compile and apply blocked patterns — DROP|DELETE|UPDATE|INSERT, XSS patterns, credential keywords. Return is_valid=False immediately on match.
  • Structural Check (1ms): Validate query length between 5–1000 chars. Detect language — accept English or Hindi only. Reject excessive special characters.
  • Domain Keyword Check (2ms): Scan for RBI banking keywords — bank, npa, fraud, dpd, penalty, circular, kyc, upi, camels, sarfaesi, inspection. If found → set is_valid=True, skip LLM.
  • LLM Fallback (150ms — only if Step 3 inconclusive): Call lightweight LLM with yes/no validation prompt. Set is_valid from response.
  • Audit Entry: Append L1 result to audit_log — query_hash, validation_layer_used, result, latency_ms.
Skip LLM for 80%+ of valid queries — domain keyword check returns immediately, saving 150ms

ASYNCIO.GATHER — RUNS SIMULTANEOUSLY
SpaCy NLP Decomposition
  • Load model once at app startup (singleton)
  • NER: extract bank names, amounts, dates, DPD values, city names, circular numbers
  • Dependency parse: identify subject + predicate + object
  • Split multi-intent queries into sub-queries list
  • Output: decomposition dict with entities, filters, sub_queries, join_hints
SLM Paraphrasing
  • Use quantized flan-t5-small (INT8) — 80ms
  • Generate Paraphrase 1: formal RBI language version for vector search
  • Generate Paraphrase 2: keyword-dense version for BM25 pre-filter
  • Output: paraphrases list — [formal_para, keyword_para]
Cache Check
  • Fingerprint query: lemmatize + sort tokens → hash
  • Check Redis query result cache
  • Hit → short-circuit: write final_response directly and jump to audit_logger
  • Miss → continue to l2_classifier
asyncio.gather() runs SpaCy + SLM + cache check simultaneously — total = 200ms not 400ms

  • Store new plan in cache (async): TTL 600s.

sql_resultrows, metadata, query_hash
audit_logappend sql tool entry
errorset if tool fails
  • Fetch schema from Redis (0ms): Load relevant table schemas, join map, and column stats. Inject into LLM prompt — only tables relevant to this query.
  • Fetch dept access rules from Redis (0ms): Get allowed_sources, denied_fields for current department.
  • LLM generates SQL (200ms): Prompt includes system rules (SELECT only), table schemas, join map, few-shot examples, decomposition entities, denied_fields list.

  • Write audit entry async: tool=nl_sql, query_hash, row_count, latency_ms, tables_accessed, dept. Non-blocking.

TOOL NODE vector_search_tool def vector_search_tool(state: RBIState) 30ms ▾
READS FROM STATE
query
paraphrasesformal + keyword versions
decompositionfor metadata pre-filter
deptdoc access filter
→
WRITES TO STATE
vector_resulttop-10 chunks + scores
audit_logappend vector entry
  • Metadata pre-filter (2ms): Filter vector index to only chunks where dept_relevance includes current user's department. If circular_ref detected in query → filter to that doc only.
  • BM25 pre-filter on keyword paraphrase (10ms): Run BM25 keyword search on filtered chunk pool using keyword-dense paraphrase. Narrow to top-50 candidates.
  • Embedding cache check (0ms): Check if query text was embedded before. Cache hit → return cached vector. Miss → call BAAI/bge-m3 embedding model (80ms).
  • Dense vector search on 50 candidates (20ms): Cosine similarity on BM25 subset only — not full 1,154 chunk index. 10x faster.
  • Hybrid scoring + rerank: Final score = 0.4 × BM25_score + 0.6 × cosine_similarity. Return top-10 chunks with scores and source metadata.
  • OCR/Image cache check: For any image or table chunks in results → check permanent OCR cache. Return cached Markdown/JSON. Never re-run OCR.
  • Write audit entry async.
Two-stage BM25 → dense search: 10x smaller search space, 90ms saved vs full index dense scan
NODE · PARALLEL parallel_executor async def parallel_executor(state: RBIState) 300ms ▾
READS FROM STATE
sourcesmultiple sources list
execution_planDAG with parallel flags
query + decomposition
→
WRITES TO STATE
sql_resultif SQL in sources
nosql_resultif NoSQL in sources
vector_resultif Vector in sources
  • Build async task list: For each source in state.sources → create async task: nl_sql_async(state), nl_nosql_async(state), vector_search_async(state).
  • Fire all tasks simultaneously via asyncio.gather(): All tool coroutines execute at same time. Total latency = slowest single tool (~300ms) not sum (~900ms).
  • Timeout guard (800ms per task): If any task exceeds 800ms → mark as TIMEOUT, continue with available results. Do not fail entire pipeline.
  • Error isolation: If one tool fails → capture error for that source only. Other sources continue. Write failure to audit_log. Validator will flag as LOW_CONFIDENCE.
  • Return all results to state: Each tool writes to its own state key — sql_result, nosql_result, vector_result independently.
Biggest latency win: 600ms saved on multi-hop queries — all 3 sources execute simultaneously

RUNS PARALLEL WITH SYNTHESIZER DRAFT
Rule Checks (10ms)
  • Row count non-zero
  • Schema type alignment
  • PII scan — Aadhaar / PAN patterns
  • Bank ID referential integrity
  • Amount sanity — no ₹0 or ₹99999Cr
  • Date range — no future dates in historical fields
  • Cross-source name consistency — bank_name matches across SQL + NoSQL
LLM Fallback (150ms — only if rules fail)
  • Hallucination check — every borrower ID traceable to source row
  • Amount plausibility relative to sector
  • Source traceability — every data point has a source citation
  • Set confidence = 0.7 if LLM validation used
  • Set confidence = 0.95 if rules-only validation
Rule-first validation (10ms) + parallel with synthesis draft — saves 200-300ms total

Streaming output: perceived latency 200ms vs 2000ms full-response wait — 10x UX improvement

TERMINAL NODE error_handler def error_handler(state: RBIState) → END 5ms ▾
  • Read error source from state: Check state.invalid_reason, state.error, state.validation_status to identify failure type.
  • Generate user-friendly error response: L1 INVALID → "This query is outside the scope of RBI supervision data. Please ask about banks, NPA, fraud, inspections, or RBI circulars." | TOOL FAIL → "Data retrieval encountered an issue. Please try rephrasing or contact support." | VALIDATION FAIL → "Result could not be verified against source data. Manual query recommended."
  • Write error audit entry: Log failure type, node that failed, error message, timestamp — for debugging and improvement.
  • Return to END.
